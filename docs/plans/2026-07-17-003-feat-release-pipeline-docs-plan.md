# feat: README refresh, GitHub About update, and Apple-Silicon-only release pipeline

## Overview

Three related pieces of polish for the newly-rebranded **Messy** app (v1.3.0, macOS/Electron):

1. **README overhaul** ŌĆö center the header, move the fork credit up near the top (below the intro), state plainly that this is a macOS Apple-Silicon-only fork of the original project (with a link), correct the stale requirements, and refresh the project-structure table.
2. **GitHub "About" update** ŌĆö refine the repo description and topics; **leave the website URL field untouched** (owner directive), with the original-repo link preserved in the README credits.
3. **Release pipeline** ŌĆö port the shape of `hungdoansy/nodl`'s release workflow into Messy, but collapsed to **Apple Silicon (arm64) only**: a single macOS-arm64 GitHub Actions job that builds, signs + notarizes (with an ad-hoc fallback when secrets are absent), generates a Gemini-authored changelog, and publishes a GitHub Release whose assets feed `electron-updater` auto-update.

## Problem Frame

Messy was just rebranded from "Mosx" and rewritten for macOS with a hardened security posture. The README still carries pre-rebrand rough edges: a left-aligned header, credits buried at the bottom, requirements that don't match `package.json` (says pnpm v9.15+, but the repo requires pnpm Ōēź11), and a project-structure table that omits files added during the security rewrite (`trust.js`, the test suite, the entitlements plist). Releases are currently produced by hand on the owner's machine (`pnpm run build`), which is slow, non-reproducible, and easy to get wrong (signing, notarization, and the `electron-updater` feed files all have to line up). The owner wants a documented, one-click release pipeline modeled on their `nodl` project but scoped strictly to Apple M chips, plus a cleaned-up public face (README + About).

## Requirements Trace

- **R1.** README header (name + icon + tagline) is horizontally centered.
- **R2.** The fork credit moves to the top of the README, immediately below the intro/tagline and before Features.
- **R3.** README states clearly that Messy is a fork of the original repo (with a working link) and is built specifically for **macOS on Apple Silicon (Apple M chips)**.
- **R4.** README requirements match reality: Node Ōēź24, pnpm Ōēź11 (currently mis-stated as pnpm v9.15+), macOS on Apple Silicon.
- **R5.** README project-structure table reflects the current source tree (adds `trust.js`, `test/`, `build/entitlements.mac.plist`; keeps existing entries accurate).
- **R6.** The GitHub repo "About" description and topics are updated to reflect the macOS/Apple-Silicon fork identity; the **website URL field is not modified**.
- **R7.** A GitHub Actions release pipeline builds Messy for **arm64 only on an Apple-Silicon runner**, produces a `.dmg` plus the `electron-updater` feed files (`latest-mac.yml`, `.dmg.blockmap`), and creates a GitHub Release with those assets attached.
- **R8.** The pipeline signs + notarizes with a Developer ID when the required secrets are present, and falls back to an ad-hoc unsigned build (still runnable locally) when they are absent ŌĆö never failing purely for lack of signing secrets.
- **R9.** Release notes are generated by porting nodl's Gemini-based changelog script (with a graceful fallback to raw git log when `GEMINI_API_KEY` is missing).
- **R10.** The version input drives `package.json` version, the tag, the artifact name, and the auto-update feed consistently; the pipeline commits the version bump + changelog and pushes the tag (nodl's flow, adapted to Messy's flat repo).

## Scope Boundaries

- **Not** changing the app's signing *model* itself (hardened runtime, fuses, entitlements) ŌĆö the pipeline consumes the existing `package.json` `build` config; it does not redesign it.
- **Not** multi-platform. No Windows/Linux/Intel targets and no build matrix ŌĆö Apple Silicon only, by directive. (nodl's 4-way matrix collapses to a single job.)
- **Not** touching the GitHub "About" **website URL** field (owner directive) ŌĆö only description/topics.
- **Not** setting up GitHub Pages or a docs site as part of this work.
- **Not** re-translating or re-branding content ŌĆö that was completed in the prior rebrand plan; this plan only *refreshes* README structure/accuracy.
- **Not** obtaining an Apple Developer account or provisioning certificates ŌĆö the pipeline is built to *use* secrets if present; creating them is an owner action outside this plan.

### Deferred to Separate Tasks

- **Actually enabling signed/notarized releases**: requires the owner to add repo secrets (Developer ID cert + App Store Connect API key). The pipeline supports it on day one but runs ad-hoc until those secrets exist ŌĆö configuring them is an owner action, documented here (Unit 5) but performed separately.
- **A `deploy-web` equivalent**: nodl has one because it's a monorepo with a web app; Messy has no web surface, so it is intentionally omitted.

### Reference Pipeline ŌĆö `hungdoansy/nodl` (public)

Files: `.github/workflows/release.yml`, `.github/scripts/generate-changelog.mjs` (plus `deploy-web.yml`, which Messy does not need).

- **Trigger:** `workflow_dispatch` with a `version` input validated as `X.Y.Z`; also checks the tag doesn't already exist (`git rev-parse "$TAG"`).
- **prepare job** (ubuntu): checkout `fetch-depth: 0` ŌåÆ `pnpm/action-setup@v4` ŌåÆ `actions/setup-node@v4` (node 22, pnpm cache) ŌåÆ lint ŌåÆ test ŌåÆ `node .github/scripts/generate-changelog.mjs` (needs `GEMINI_API_KEY`) ŌåÆ outputs `version`, `tag`, `changelog`.
- **build job:** 4-way OS/arch matrix; each: install ŌåÆ set version in `package.json` ŌåÆ test ŌåÆ `npx electron-builder --${{ matrix.arch }} --publish never` (with `CSC_IDENTITY_AUTO_DISCOVERY: false`) ŌåÆ upload per-platform artifacts.
- **release job** (ubuntu): `actions/download-artifact@v4` (merged) ŌåÆ inject changelog into `CHANGELOG.md` as `## [tag] (date)` ŌåÆ commit `release: $TAG` ŌåÆ `git tag` + `git push origin main` + push tag ŌåÆ `softprops/action-gh-release@v2` (name/tag from prepare, body = changelog, files = artifact globs) ŌåÆ `geekyeggo/delete-artifact@v5` cleanup.
- **`generate-changelog.mjs`:** finds latest `v*` tag, `git log --no-merges --pretty=tformat:%s` since it (or last 20 commits if none), filters `release:`/`docs:` commits, strips conventional-commit prefixes, formats bullets. If `GEMINI_API_KEY` set, POSTs to `generativelanguage.googleapis.com/v1beta/models/${GEMINI_MODEL:-gemini-2.5-flash}:generateContent` to rewrite as user-facing notes; on missing key or API error, **falls back to the raw list and still exits 0**. Writes `text` to `GITHUB_OUTPUT`; appends a summary to `GITHUB_STEP_SUMMARY`.

**How Messy differs from nodl (the adaptation):** single arm64 job (no matrix, no prepare/build/release split needed ŌĆö one macOS-arm64 job can do all of it, or keep a light prepareŌåÆbuildŌåÆrelease split for clarity); Node 24 not 22; repo-root paths not `apps/desktop/**`; signing is **conditional** (nodl always disables it) ŌĆö Messy signs + notarizes when secrets exist; and the release **must** include the `electron-updater` feed files (`latest-mac.yml`, `*.dmg.blockmap`), which nodl's desktop release also needs but which are easy to drop when collapsing the workflow.

### External References

- Reference pipeline: `https://github.com/hungdoansy/nodl` (`.github/workflows/release.yml`, `.github/scripts/generate-changelog.mjs`).
- Upstream/original repo (kept in README credits, R3): `https://github.com/nct88/Messenger-Win` (renamed `https://github.com/nct88/MessengerMulti-Windows`).
- GitHub-hosted macOS runners are Apple Silicon: `macos-14` (M1) and `macos-15`/`macos-latest` (arm64). Pin an explicit arm64 image for reproducibility.
- `electron-builder` macOS signing/notarization env: `CSC_LINK` (base64 `.p12`) / `CSC_KEY_PASSWORD` (Developer ID cert), `APPLE_API_KEY`/`APPLE_API_KEY_ID`/`APPLE_API_ISSUER` (notarytool), `CSC_IDENTITY_AUTO_DISCOVERY=false` (force ad-hoc). Confirm exact notarize toggle against `electron-builder@26.15.3` docs at implementation.
  - **`APPLE_API_KEY` is a filesystem path to the `.p8` key file, not the key contents** (feasibility-review finding). Apple's `.p8` can't be used directly from a plain secret ŌĆö store it base64-encoded in a secret and **decode it to a temp file** in the workflow, then set `APPLE_API_KEY=/path/to/AuthKey.p8`. `_ID`/`_ISSUER` are plain string secrets.
  - `CSC_LINK` (base64 `.p12`) + `CSC_KEY_PASSWORD` is sufficient on its own ŌĆö **electron-builder imports the cert into a temporary keychain automatically**; no manual `security create-keychain`/`import-codesign-certs` step is needed.
  - With App Store Connect API-key auth, `-c.mac.notarize=true` (boolean) is enough ŌĆö `teamId` is only required for Apple-ID auth, so it is correctly omitted here.
  - `latest-mac.yml` + `.dmg.blockmap` are generated into `dist/` **because a `build.publish` block is present** in `package.json`; `--publish never` suppresses only the *upload*, not the metadata generation. Keeping the `publish` block (Unit 4) is what makes the softprops-attached feed valid.

## Key Technical Decisions

- **Apple-Silicon-only, single job (no matrix):** run on a pinned Apple-Silicon runner (`macos-14`) and build `--arm64` only. This is the literal interpretation of "built for Apple M chip only" and drops nodl's cross-platform matrix entirely. A macOS runner can also do the tag/release git work, so the prepare/build/release split is optional ŌĆö keep a light split only if it aids readability; a single job is acceptable given one target.
- **Conditional signing with ad-hoc fallback (R8, owner choice "Both, with fallback"):** a setup step detects whether signing secrets are present. If yes ŌåÆ import the Developer ID cert and build with notarization enabled. If no ŌåÆ export `CSC_IDENTITY_AUTO_DISCOVERY=false` and build ad-hoc. The build step never hard-fails for missing secrets. **Consequence to document:** ad-hoc CI builds run locally but are Gatekeeper-flagged for other users and **cannot** auto-update (signature verification) ŌĆö this matches the existing README note; auto-update becomes real only once secrets are added.
- **Notarization is opt-in via env, not hard-coded:** `package.json` keeps `build.mac.notarize: false` as the safe default for local/ad-hoc builds; CI enables notarization only in the signed branch (e.g., via an electron-builder `-c.mac.notarize=...` override or an env-gated config). Exact toggle syntax for `electron-builder@26.15.3` is deferred to implementation.
- **Release assets must include the updater feed (R7):** build with `--publish never`, then attach the **full** `dist/` output that `electron-updater` needs ŌĆö `Messy-<version>-arm64.dmg`, `Messy-<version>-arm64.dmg.blockmap`, and `latest-mac.yml` ŌĆö via `softprops/action-gh-release@v2`. Missing `latest-mac.yml`/blockmap silently breaks auto-update. (Alternative considered: `electron-builder --publish always`, which uploads the feed and creates the release itself using `GH_TOKEN`; rejected as the primary path to stay close to nodl's softprops + AI-changelog shape, but noted as a simpler fallback if asset globbing proves fragile.)
- **Port the Gemini changelog verbatim in behavior, de-nested in paths (R9):** same commit-collection, prefix-stripping, Gemini call, and missing-key/error fallback; only the working directory changes from `apps/desktop` to repo root, and the `CHANGELOG.md` inject/commit targets root files.
- **`workflow_dispatch` with `X.Y.Z` version input (R10):** matches nodl and gives the owner explicit control over version/tag; the input becomes `package.json` `version`, the `vX.Y.Z` tag, the `Messy-<version>-arm64.dmg` name, and the updater feed version ŌĆö all from one source. (Tag-push trigger considered and not chosen: less explicit, and the workflow itself pushes the tag.)
- **Node 24 / pnpm 11 in CI:** align with `engines` + `.nvmrc` + `packageManager` (nodl uses Node 22; Messy requires Ōēź24).
- **About: description + topics only, website untouched (R6, owner choice):** update via `gh api` PATCH; the original-repo link stays in README credits, so the About website field is deliberately left as-is.

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

Release flow (single Apple-Silicon runner shown collapsed; a prepareŌåÆbuildŌåÆrelease split is an equivalent structuring choice):

```mermaid
flowchart TD
    A[workflow_dispatch: version=X.Y.Z] --> B{version matches X.Y.Z<br/>and tag vX.Y.Z absent?}
    B -- no --> X[fail fast with message]
    B -- yes --> C[checkout fetch-depth:0 +<br/>pnpm@11 + node 24 + install]
    C --> D[pnpm test]
    D --> E[set package.json version = X.Y.Z]
    E --> F[node .github/scripts/generate-changelog.mjs<br/>ŌåÆ changelog text GITHUB_OUTPUT]
    F --> G{signing secrets present?}
    G -- yes --> H[import Developer ID cert;<br/>build --arm64 --publish never + notarize]
    G -- no --> I[CSC_IDENTITY_AUTO_DISCOVERY=false;<br/>build --arm64 --publish never ad-hoc]
    H --> J[dist/: .dmg + .dmg.blockmap + latest-mac.yml]
    I --> J
    J --> K[commit 'release: vX.Y.Z' + CHANGELOG.md;<br/>tag + push main + push tag]
    K --> L[softprops/action-gh-release@v2<br/>body=changelog, files=dist/*.dmg,*.blockmap,latest-mac.yml]
```

Directional signing-gate sketch (not final syntax):

```yaml
# choose signing mode from secret presence
- id: signmode
  run: |
    if [ -n "${{ secrets.CSC_LINK }}" ]; then echo "signed=true" >> "$GITHUB_OUTPUT";
    else echo "signed=false" >> "$GITHUB_OUTPUT"; fi
- name: Build (signed)
  if: steps.signmode.outputs.signed == 'true'
  env: { CSC_LINK, CSC_KEY_PASSWORD, APPLE_API_KEY, APPLE_API_KEY_ID, APPLE_API_ISSUER }
  run: npx electron-builder --mac dmg --arm64 --publish never -c.mac.notarize=true
- name: Build (ad-hoc)
  if: steps.signmode.outputs.signed == 'false'
  env: { CSC_IDENTITY_AUTO_DISCOVERY: "false" }
  run: npx electron-builder --mac dmg --arm64 --publish never
```

## System-Wide Impact

- **Interaction graph:** the release workflow consumes `package.json` `build`/`publish` config and the `dist/` output contract; the changelog script consumes git history and (optionally) the Gemini API; the About change touches repo settings only.
- **API surface parity:** the `electron-updater` feed (`latest-mac.yml` + blockmap) is a real external contract ŌĆö the Release assets must match what installed clients poll for at `hungdoansy/messy`. Dropping a feed file silently breaks updates.
- **Error propagation:** the changelog script must never fail the release for a missing/failed Gemini call (fallback to raw log); the signing gate must never fail the build for missing secrets (fallback to ad-hoc).
- **State lifecycle risks:** the workflow commits a version bump + changelog and pushes `main` + a tag ŌĆö a failed mid-run leaves a partial state (e.g., committed bump, no release); prefer building/validating *before* the commit/tag/push, mirroring nodl's ordering.
- **Unchanged invariants:** the app's security config (`hardenedRuntime`, `electronFuses`, entitlements, `trust.js` allowlist), the `publish` target, and local `pnpm run build` behavior are all preserved; only CI adds a *signed* path on top.

| Risk | Mitigation |
|------|------------|
| Ad-hoc CI builds can't auto-update and are Gatekeeper-flagged for other users | Documented explicitly (README + Unit 5); signed path activates the moment secrets are added; matches existing README note |
